# Design: 提升公开站点可读性与足球分析表达

## 核心原则

1. **比赛先于数据表**：普通用户进入后先看到比赛、队伍、比分/时间、状态，而不是数据库条数。
2. **结论有边界**：展示赛果、市场隐含概率、风险和证据，不生成下注建议。
3. **图表必须可读**：每个图表都有文字结论、数值标签和空状态，不依赖颜色表达含义。
4. **缺失也要可解释**：没有 xG、伤停、赔率或球队画像时，不显示空白大卡，而是说明“缺什么、为什么影响判断”。
5. **中国用户语境**：优先中文译名；无法确认的球员名保留原文；术语提供中文解释。

## 页面设计

### 首页 `/`

首屏结构改为：

- 顶部一句话：`今天先看这几场：比分/开球时间、数据准备度、主要风险。`
- 焦点比赛区：
  - 主队国旗 + 中文名 + FIFA code。
  - 客队国旗 + 中文名 + FIFA code。
  - 中间显示比分或开球时间：
    - 完赛：`5 - 0`，并突出胜方。
    - 未开赛：`今晚/今日 待开球` 或具体时间。
  - 状态 chip：未开赛、进行中、完赛、待同步。
  - 三个用户化指标：`数据准备度`、`风险`、`证据`。
- 新手解释区：
  - `比分/赛果`：已完赛时最重要。
  - `数据准备度`：表示资料是否够完整，不等于胜率。
  - `风险`：伤停、天气、赛程、舆情等可能改变判断。
- 底部入口保留，但从编号卡改为任务卡：`看比赛`、`看赛前分析`、`查证据`、`看球队/球员`。

### 比赛中心 `/evidence/matches`

- 左侧列表改为 scoreboard card：国旗、队名、比分/时间、状态、证据数、冲突数。
- 详情顶部增加 match hero：
  - 大比分牌。
  - 胜方高亮；平局显示平局 badge。
  - 场地、阶段、比赛日。
- 新增视觉模块：
  - `进球时间线`：根据 `match_team_stats.scoringMinutes` 和进球事件渲染。
  - `球队对比条`：进球、失球、事件数、证据数；以后可接 xG、射门等。
  - `事件时间线`：按分钟分组，进球/红黄牌/换人使用统一 SVG 图标。
- 证据链按来源可信度排序，显示“支撑比分/阵容/事件/场地”。

### 赛前作战室 `/workbench`

- 当前比赛顶部用国旗 + 比赛卡替代纯文本标题。
- `decisionSteps` 改为用户语言：
  - `资料够不够`、`球队/球员有没有画像`、`赔率市场有没有数据`、`外部因素有没有风险`、`有没有分析报告`。
- 增加图表：
  - 数据完整性 bullet/progress。
  - 胜平负隐含概率条：从赔率 selections 派生，只展示市场含义。
  - 舆情风险矩阵：风险等级 × 分类。
  - 证据可信度条：按 reliabilityScore 分桶。
- 增加术语说明：xG、PPDA、隐含概率、完整度、冲突、过期。

### 舆情与外部因素 `/evidence/sentiment`

- 顶部从“因素 203”改为：`当前主要风险：伤停/天气/裁判/赛程/市场热度`。
- 分类改中文可读分组：
  - 天气与场地
  - 裁判与判罚倾向
  - 伤停与训练
  - 赛程、旅行与休息
  - 战意、轮换与更衣室
  - 主帅发布会与球队新闻
  - 媒体/球迷舆情
  - 盘口/公众投注/市场热度
  - 历史交锋与战术对位
- 每条 factor 显示：影响方向、风险等级、时效、来源可信度、建议核查动作。

### 球队画像 `/evidence/teams`

- 列表卡显示国旗、国家/地区、小组、球员覆盖、事实覆盖。
- 详情顶部是球队名片：国旗、中文名、FIFA code、洲别、小组、画像完整度。
- 攻防画像空值显示“待补采：缺少进攻/防守结构化画像”。
- 图表：近期进失球、首球时间、球员位置覆盖；未来接 xG/xGA/PPDA。

### 球员画像 `/evidence/players`

- 列表支持按球队、位置、状态筛选。
- 详情顶部显示：姓名、球队国旗、位置、号码、状态。
- 可用性卡：伤病、红黄牌、训练、预计出场，缺失时显示待补采。
- 事实卡按“伤停/训练/表现/纪律/更衣室/媒体”分组。

## 后端设计

### Public DTO 扩展

新增/扩展 DTO 时保持兼容：原字段不删除，新字段可选。

建议新增通用结构：

```java
record PublicTeamVisual(
    Long teamId,
    String teamName,
    String fifaCode,
    String countryIso2,
    String flagUrl,
    String countryRegion
) {}

record PublicScoreboard(
    Integer homeScore,
    Integer awayScore,
    String scoreDisplay,
    String winnerSide,
    String resultText,
    String scoreSource
) {}

record PublicVisualMetric(
    String label,
    BigDecimal value,
    String unit,
    String tone,
    String explanation
) {}
```

### 比分派生规则

1. 优先读取 `match_team_stats` 中主队和客队 `goalsFor/goalsAgainst`。
2. 若缺失，则从 `source_evidence.summary` 中可识别的比分文本作为弱来源，只用于展示并标记 `scoreSource=EVIDENCE_TEXT`。
3. 若仍缺失：
   - 完赛但无比分：显示 `比分待核对`。
   - 未开赛：显示开球时间或 `待开球`。
4. 胜负只在明确比分时派生：`HOME_WIN`、`AWAY_WIN`、`DRAW`、`UNKNOWN`。

### 国旗与国家元数据

短期：前端用 `fifaCode -> iso2` 映射表和 flag URL/本地 flag key 渲染，缺失时显示 FIFA code。

长期：数据库 teams 增补：

- `country_iso2`
- `flag_asset_key`
- `confederation`
- `group_name`
- `metadata_source_ref`

### 图表数据聚合

- `teamComparison`：从 teamStats、events、evidence、oddsSelections 聚合。
- `oddsImpliedProbability`：使用现有 `impliedProbability`，缺失时不前端反推。
- `riskDistribution`：按 riskLevel、factorCategory 聚合。
- `evidenceQuality`：按 reliabilityScore 分为高/中/低/未评分。

## 数据导入设计

### SOURCE JSON 扩展建议

支持以下结构，批准后入库到现有舆情/证据表或未来指标表：

- `weather`：温度、降雨、风速、湿度、草皮、来源。
- `referee`：裁判姓名、场均黄牌、红牌、点球、执法倾向。
- `travel_rest`：休息天数、旅行距离、跨时区、连续客场。
- `motivation`：出线形势、轮换动机、必须取分程度。
- `injuries`：球员、伤病、预计回归、影响等级、来源。
- `market_signals`：盘口初盘/即时、变化时间、公众投注、资金比例。
- `advanced_metrics`：xG、xGA、npxG、PPDA、xPTS、射门质量、渐进传球。

## 测试策略

- 后端：
  - 比分派生测试：teamStats 完整、teamStats 缺失/evidence 有比分、完赛无比分、未开赛。
  - 国旗/国家元数据 DTO 兼容测试。
  - public API 脱敏回归测试。
- 前端：
  - 组件测试：ScoreboardCard、FlagTeamName、MetricBar、GlossaryHint。
  - 页面测试：首页/比赛/作战室显示比分、国旗、胜负/待开球、空状态解释。
  - 可访问性：图表必须有文本摘要；色彩不是唯一语义。
- 浏览器：
  - PC 1440×900 首屏截图。
  - 移动 375px 宽度基本可读，无横向滚动。

## 实施顺序建议

1. 第一轮：不新增数据库表，先用现有数据派生比分、胜负、国旗、基础图表和新手解释。
2. 第二轮：后端 public DTO 正式增加 visual summary，补测试。
3. 第三轮：数据库国家元数据和指标快照扩展。
4. 第四轮：导入模板和示例 JSON，补充球队/球员/舆情数据。

## 第二轮补充：证据中心 / 赔率中心

### 问题

赔率中心仍偏后台快照列表：用户能看到公司、玩法、盘口，但看不到“这场是谁对谁、比分/状态是什么、市场倾向到底偏哪边、概率条如何理解”。对不懂足球的新用户来说，赔率数字仍是孤立数据。

### 设计

- 公开赔率列表和详情页都补充 `PublicTeamVisual` 与 `PublicScoreboard`。
- 左侧快照列表使用紧凑比赛卡，显示国旗、队名、比分/待开球，再显示公司/玩法/盘口。
- 详情页顶部使用完整比赛卡，避免用户在赔率页迷失比赛上下文。
- 当前玩法选项增加隐含概率条，直接呈现“市场更偏向谁/哪个方向”，并明确说明这只是市场信号，不是胜率和下注建议。
- 空数据必须显示“待补采”：缺盘口、缺实时变动、缺公众投注、缺资金比例、缺伤停/裁判等，不展示空白图表。

## 第三轮补充：舆情中心首屏与外部因素覆盖矩阵

### 问题

舆情中心虽然已经按因素和风险展示，但 PC 首屏仍容易被普通用户理解成“后台因素列表”：用户还需要自己判断这些舆情属于哪场比赛、比赛是否已开、比分是什么，以及哪些足球分析维度还没有数据。

### 设计

- 公开舆情列表和比赛详情补充 `PublicTeamVisual` 与 `PublicScoreboard`，让每条外部因素都绑定到主客队、国旗和比分/待开球状态。
- 舆情页 Hero 右侧直接显示当前选中比赛的小比分卡；首屏就能看到“谁对谁、比分/状态”。
- Hero 风险文案避免把“转播平台/其他”误说成主要风险；当没有高风险评分时，改为说明“暂无高风险评分 + 已覆盖哪些维度 + 哪些维度待补采”。
- 增加 9 类外部因素覆盖矩阵：天气/场地、裁判、伤停/训练、赛程旅行休息、战意轮换、更衣室/发布会、媒体球迷、盘口公众投注、历史交锋战术对位。
- 覆盖矩阵必须显示“已覆盖几条”或“待补采”，缺失维度不展示空图表，避免用户误以为数据完整。

## 第四轮补充：比赛证据质量面板

### 问题

比赛详情已有证据链和冲突状态，但仍偏“后台列表”：用户能看到来源和可信度数字，却不知道这条证据到底支撑比分、阵容、场地、裁判、天气、盘口还是背景信息，也不知道证据是否过期、是否足够用于判断。

### 设计

- 公开比赛证据 DTO 增加派生展示字段：
  - `qualityLevel`：按 `reliabilityScore` 分为高可信/中可信/低可信/未评分。
  - `freshnessStatus`：按 `evidenceTime` 分为新鲜/需复核/已过期/时间待同步。
  - `supportsConclusion`：从来源类型、来源引用和摘要中归类为比分赛果、阵容首发、场地赛程、裁判判罚背景、天气外部因素、市场赔率信号、赛程官方确认或背景证据。
  - `suggestedAction`：给出“可作为核心证据/需要第二来源/仅弱参考/需人工补评/证据过期需重查”等核查动作。
- 比赛中心在比分和事件图表后增加“证据质量面板”：
  - 左侧给新手一句话解释当前证据是否足够。
  - 中间用条形图展示高/中/低/未评分证据分布，图表有文字标签，不只靠颜色。
  - 右侧按“支撑了什么结论”分组，显示每类证据数量和代表来源。
  - 人工核查清单固定提醒：先核比分与胜负，再核阵容、伤停、场地、裁判、天气和市场信号。
- 证据链卡片显示质量 chip、时效 chip、支撑结论和核查动作。
- 冲突卡片只展示公开状态，不暴露当前值/新值/原始 payload，但必须提示“对照正式来源人工核查”。

## 第五轮补充：球队/球员画像健壮性

### 问题

球队/球员画像已经有国旗、名单、事实、阵容和补采提示，但普通用户仍需要自己判断“这个画像到底够不够看”。尤其是球员页，如果没有球队国旗、预计首发、伤病、停赛、训练负荷和 xG/xA 等指标，很难理解一个球员会怎样影响比赛。

### 设计

- 公开画像详情增加统一 `readiness`：
  - `score`：0-100 健壮性分数。
  - `level`：较健壮/部分健壮/偏薄弱。
  - `summary`：给新手看的单句解释。
  - `strengths`：已具备的维度。
  - `missingDimensions`：还缺什么。
  - `nextActions`：下一步补采建议。
- 球队画像健壮性维度：
  - 国家/国旗/FIFA 元数据。
  - 攻击画像、防守画像。
  - 球员名单覆盖。
  - 画像事实数量。
  - 阵容/首发线索。
  - 近期进失球或历史趋势。
  - 外部因素。
  - 证据链。
  - 高阶指标。
  - 冲突作为扣分项。
- 球员画像健壮性维度：
  - 所属球队/国旗。
  - 号码和位置。
  - 基础状态、伤病状态、红黄牌状态、更衣室状态。
  - 画像事实数量与状态/训练事实。
  - 最新高阶指标。
- 公开画像详情增加 `latestMetric`：
  - 球队：xG、xGA、npxG、PPDA、xPTS、射门、控球、渐进传球、定位球 xG、状态分。
  - 球员：出场分钟、进球、助攻、xG、xA、npxG、射门、关键传球、训练负荷、可用性、预计首发概率。
- 前端球队/球员详情首屏新增“新手一眼判断”卡：总分、等级、已具备、还缺什么。
- 前端高阶指标卡必须有文字解释和空状态，避免只展示数字或空白图表。
- 球员列表和详情尽量显示所属球队国旗，避免球员数据脱离国家/球队上下文。

## 第六轮补充：球员列表可检索性与 PC 长列表性能

### 问题

球员画像页已经补充球队国旗、健壮性和高阶指标，但公开库球员数量可能达到数百到上千。PC 端左侧一次性渲染全部球员时，普通用户仍会陷入“名单墙”：很难快速找到某队、某位置、某个状态的球员，也会把右侧详情列撑出很长页面。

### 设计

- 球员列表首屏必须提供搜索和筛选：
  - 搜索：姓名、球队、FIFA code、号码、位置、状态。
  - 球队筛选：按所属球队过滤。
  - 位置筛选：门将/后卫/中场/前锋等。
  - 状态筛选：可出场、伤停、停赛、出战存疑等。
- 左侧列表只渲染当前页球员，默认每页 24 人，避免 50+ 长列表造成滚动和渲染压力。
- PC 端筛选结果放入可聚焦的受限高度列表容器，筛选区和分页区保留在同一侧栏内，避免左侧名单把右侧详情列撑成长页面。
- 筛选结果需要明确显示“已筛出 X / Y 名球员”，空结果给出恢复路径，而不是空白。
- 所有筛选控件使用原生输入/下拉，保证键盘可达、焦点可见、触控区域不小于 44px。
- 详情列保持当前选中球员，不因用户试探性搜索而频繁请求详情；用户点击筛选结果后再切换详情。

## 第七轮补充：舆情外部情报补采优先级

### 问题

舆情页已经有 9 类覆盖矩阵，但普通用户仍然需要自己判断“本场外部情报够不够”。如果当前正式库只覆盖天气或场地，页面虽然显示了“待补采”，但缺少优先级：用户不知道应该先补伤停、裁判、赛程旅行、市场热度还是战术对位。PC 端左侧因素列表也可能因记录很多而拉长页面。

### 设计

- 舆情筛选区增加搜索，支持按比赛、因素、来源和摘要定位线索。
- 左侧因素列表分页，每页只渲染 16 条，并放入可聚焦限高滚动容器，避免 PC 端列表撑高右侧详情。
- 详情页在比分卡下新增“本场外部情报完整度”：
  - 按 9 类外部因素计算覆盖分。
  - 显示已覆盖维度和仍缺维度。
  - 高优先级缺口包括伤停训练、裁判判罚、赛程旅行休息、战意轮换和盘口/公众投注/市场热度。
- 新增“补采优先级清单”，每个缺口都解释为什么影响判断、下一步需要补什么。
- 页面文案明确边界：参考足球 AI 研究常见框架（xG/高级指标、伤病、裁判、盘口变动、公众投注、球队新闻、赛程旅行、天气、历史/战术对位），但公开站点只做数据解释和核查清单，不生成下注建议。

## 第八轮补充：首页新手赛事情报入口

### 问题

首页已经把公开概览、焦点比赛、风险、证据、赔率和决策入口整合在一起，但对完全不懂足球的新用户仍然偏“数据看板”：用户先看到多个 KPI 和模块入口，却不知道今天应该先看哪场比赛、比分/胜负是什么、数据够不够、风险应该先核哪类。

### 设计

- 首页在 KPI 前新增“新手赛事情报入口”，把第一场公开焦点比赛提升为首屏行动对象。
- 入口左侧继续使用 `ScoreboardCard`，保留国旗、主客队、比分/待开球、胜负、资料完整度和风险数，确保比分与胜负成为阅读锚点。
- 当正式库暂缺 FIFA code 时，前端可基于常见中文国家名临时推断国旗；这只作为展示兜底，长期仍应由数据库国家元数据提供权威代码。
- 中间用“三步判断”卡直接回答：
  - 比分/赛果：显示比分、主队胜/客队胜/平局/待开球。
  - 数据准备度：把完整度分翻译为“较完整/基础可读/偏薄弱”。
  - 风险先看：把风险数翻译为伤停、盘口、天气、裁判、来源冲突等核查动作。
- 右侧补充条形图：
  - 数据准备度条：帮助普通用户快速判断这场是否值得继续读。
  - 风险压力条：帮助用户理解“风险多不多”，并配套文字解释，避免只靠颜色。
- 下方阅读顺序固定为“先确认比分/状态 → 再看数据是否够用 → 最后核对风险”，让非足球用户也能按流程进入比赛中心。
- 页面继续坚持非下注建议边界：展示数据解释和核查清单，不输出投注方向。

## 第九轮补充：比赛中心胜负强对比

### 问题

### 设计

- 比赛详情顶部、紧邻比分卡的位置新增“胜负强对比”：
  - 直接显示胜方/平局/待开球/比分待核对。
  - 同屏展示比分、赛果、证据数、冲突数，让结果与可信度绑定。
- “两队强弱条”使用横向条形对比，优先比：
  - 比分/进球：进球更多的一方更直观占优。
  - 失球压力：失球更少的一方防守结果更稳。
  - 首球时间：首球更早的一方更早打开局面。
- 若球队统计或事件缺失，显示“待补采”，不把 0 当真实弱势，避免误导。
- 右侧固定“新手阅读顺序”：
  1. 先看比分结论。
  2. 再看两队强弱条。
  3. 最后核证据质量与冲突。
- 条形图必须带文字值与 aria-label；色彩只做辅助，不能作为唯一含义。
- 该面板只解释公开数据，不输出投注建议。

- 桌面侧栏和移动底部导航的“证据中心”统一进入 `/evidence`，再由总览页分流到赛程、赔率、舆情、球队和球员画像。
- 总览首屏展示：
  - 公开数据可读性总分，综合比分、赔率、外部因素、球队画像和球员状态覆盖。

- “足球 AI 分析数据地图”明确补充高阶指标、伤停训练、裁判、盘口变动、公众投注、球队新闻、赛程旅行、天气、历史交锋和战术对位等维度；页面只做数据解释和核查路径，不提供投注方向。
- 总览页只调用公开列表 API，不主动拉详情 API，避免入口页加载过重，也减少公开敏感字段暴露面。

  - 结论类型筛选使用原生 `select`。

  - 显示“已筛出 X / Y 份报告”，空结果提供清除筛选。

## 第十六轮补充：球员画像全局健壮度与公开敏感词回归

### 问题

球员画像页虽然已有单个球员的“新手一眼判断”和筛选分页，但当正式库一次性展示 1000+ 球员时，新用户仍然先看到名单和数字：不知道这批球员资料整体是否够健壮，也不知道应该先补画像事实、伤停牌面、训练负荷还是高阶指标。浏览器巡检还发现球员列表分页按钮低于 44px，赔率页公开边界文案虽然是在说明“不展示”，但仍直接出现资金、票号、投注金额等敏感词，容易破坏公开脱敏检查。

- 球员画像页在 KPI 后新增“球员画像健壮度总览”：
  - 综合画像事实覆盖、可用性线索、位置覆盖和球队归属给出全局资料健壮度分。
  - 明确说明该分数不是球员强弱，而是公开资料是否够看。
  - 用条形图展示画像事实、可用性、位置、球队归属覆盖度，配套文字说明缺口数量。

- 球员列表分页按钮与筛选恢复按钮统一保持 44px 可达，保证 PC 与移动端都不出现小点击目标。
- 赔率页公开边界文案避免直接展示敏感执行名词，改为“个人执行明细”；盘口缺口提示中的资金比例改为市场热度比例，保持分析维度但不暴露执行语义。
- 保持公开边界：页面只解释资料覆盖、赔率快照、盘口和隐含概率，不输出操作方向。

## 第十七轮补充：球队画像全局健壮度与补采优先级

### 问题

球队画像详情已经能展示单队“新手一眼判断”、国旗队名、攻防画像和高阶指标，但公开球队页顶部仍只给球队/球员/事实/证据计数。普通用户无法先判断“这一批球队资料整体够不够看”，也不知道要先补名单、画像事实、攻防风格，还是国家/国旗/小组元数据。正式库扩到 48 支球队后，用户容易被列表和详情卡片分散注意力。

## 第四十轮补充：公开比分派生链路补强

### 问题

前端已经把比分牌、胜负强对比和比分状态总览作为新手阅读锚点，但后端公开 DTO 仍主要依赖 `match_team_stats` 或证据摘要文本来给出比分。若正式库已经入库了进球时间线（`match_events`），但尚未补齐球队统计或证据摘要没有写明“2-1”，公开页仍可能显示“比分待核对”。对完全不懂足球的新用户来说，这会直接破坏“先看比分/胜负”的核心路径：明明事件里有进球，却看不到比分和胜负。

### 设计

- 公开比分统一按以下优先级派生：
  1. `match_team_stats.goals_for`：官方/统计口径优先，作为最权威的结构化比分。
  2. `match_events` 进球事件：当球队统计缺失时，按主客队进球事件数兜底生成比分。
  3. `source_evidence.summary` 文本比分：仅在结构化统计和事件比分都缺失时，从证据摘要中解析。
  4. 无法派生时继续显示“待开球 / 比分待核对 / 待同步”。
- `match_events` 兜底只统计 `event_type = GOAL` 或 `event_type LIKE GOAL_%` 的事件，避免把黄牌、红牌、换人等非进球事件误算为比分。
- 本轮暂不把 `OWN_GOAL` 自动计入任一方，避免在缺少受益方字段时把乌龙球错误归给事件所属球队；乌龙球应后续通过更明确的数据模型或人工核查补采。
- 覆盖所有会展示比分卡的公开后端入口：比赛中心、首页概览、赛前作战室、赔率中心、舆情与外部因素中心，确保用户从任一入口进入都能看到一致的比分和 `scoreSource=MATCH_EVENTS`。
- 补充后端回归测试，分别覆盖列表和详情接口在缺少 `match_team_stats`、证据摘要无比分但存在进球事件时，仍能输出比分、胜负方和事件比分来源。
- 本轮只增强公开读模型的派生链路，不改变数据库权威来源、不把事件比分写回正式统计表、不读取原始 payload，也不输出操作方向。

- 仍不自动把 `OWN_GOAL` 计入任一方，避免在缺少受益方字段时把乌龙球归属算错。
- 比赛中心、首页概览、赛前作战室、赔率中心和舆情中心统一使用扩展后的口径。

## 第八十五轮补充：球队球员画像正式库事实派生

### 问题

真实运行库中 48 支球队、1248 名球员已具备国家队元数据、名单、阵容、进失球和赛程记录，但公开画像接口只统计已审核的 `team_profile_facts` 与 `player_profile_facts`。当审核事实表为空时，球队/球员页面顶部会显示画像事实为 0，详情区也缺少可直接核对的基础材料，导致已有正式库字段没有转化为页面可见的事实卡片。

### 设计

- 公开球队画像在保留已审核事实的基础上，从正式库只读派生基础事实：国家队上下文、基础画像字段、名单覆盖、阵容记录、近期比分、赛程赛果、外部因素字段与高阶指标字段。
- 公开球员画像在保留已审核事实的基础上，从正式库只读派生基础事实：国家队归属、名单身份、可用性字段与表现指标字段。
- 列表汇总 `factCount` 同步计入上述派生事实，使顶部统计、筛选和详情事实数量口径一致。
- 派生事实标记来源为“正式库派生”，`sourceRef` 指向正式表或字段名；不读取、不输出 raw payload，不把待审核 JSON 纳入公开口径。
- 保持已审核事实优先展示，派生事实作为正式库字段摘要补充；不改变证据数量口径，避免把字段摘要误计为外部证据链。
- 补充后端回归测试，覆盖无已审核事实时仍能从正式表生成球队/球员事实，并继续排除未审核事实内容。

## 第八十六轮补充：舆情外因正式库派生覆盖

### 问题

舆情页已有天气、场地、裁判和转播平台等正式入库因素，但 9 类外部因素矩阵仍只识别到天气/场地与裁判两个维度。与此同时，正式库的赛程时间字段和赔率市场快照已经覆盖全部比赛，却没有进入公开外因中心，导致赛程与市场维度被显示为缺失。

### 设计

- 公开舆情总览在保留 `match_context_factors` 已入库因素的基础上，只读派生两类正式库因素：
  - `SCHEDULE/MATCHDAY`：由 `matches` 的比赛日、开球时间、赛事、阶段和场地字段生成赛程时间因素；
  - `MARKET_SIGNAL/{market_code}`：由 `odds_market_snapshots` 的公司、市场、快照类型、市场线、选项数和采集时间生成市场快照因素。
- 单场舆情详情同步追加上述派生因素，但排序保持已入库因素优先，派生因素作为补充材料展示。
- `categories` 接口同步补充 `SCHEDULE` 与 `MARKET_SIGNAL`，使前端筛选与 9 类外因矩阵能识别正式库派生维度。
- 派生因素统一标记 `sourceName` 为正式表来源、`sourceRef` 指向 `matches` 或赔率快照来源；不读取、不输出 raw payload，不纳入待审核 JSON。
- 市场派生文本只呈现公开市场字段和采集状态，不提供投注、下注、金额或操作方向。
- 顶部主要风险文案按当前已覆盖与缺失维度动态生成，避免赛程或市场已覆盖后继续显示固定缺口。
- 补充后端回归测试，覆盖总览、详情和分类接口中的赛程与市场派生因素。

## 第九十轮补充：首页无未来赛程时的最近比赛回退

## 第九十一轮补充：比分卡状态与已出比分一致

### 问题

- 比赛中心的状态分组同步把已有明确比分的比赛纳入已出比分口径，避免“只看待开球”筛选混入有胜负结果的比赛。

## 第九十二轮补充：舆情页比分证据兜底与其它公开页一致

### 问题

真实运行库中同一场比赛可能同时存在多条 `source_evidence`。公开比赛中心、首页和作战室用最高可信来源摘要解析比分兜底，但舆情公开服务使用最新来源摘要。当最新来源摘要不含比分、较高可信来源摘要含有明确比分时，舆情页会继续显示赛前状态，和其它公开页的 `3 - 0`、胜负结果不一致。

### 设计

- 舆情公开服务的比赛查询、外因汇总和比赛视觉材料统一按 `source_evidence.reliability_score DESC, id DESC` 选择来源摘要，保持与比赛中心、首页和作战室一致。
- `scoreboard` 的 `homeScore`、`awayScore`、`winnerSide`、`resultText` 与 `scoreSource` 继续复用既有比分解析链路；当正式比分、事件比分缺失时，最高可信来源摘要可作为 `EVIDENCE_TEXT` 兜底。
- 舆情总览、单场详情、赛程派生外因和市场派生外因共享该比赛视觉口径，避免同一页面内赛前状态与已有比分记录并存。
- 该改动只读正式来源证据摘要，不读取、不输出 raw payload，不改变原始比赛状态字段。
- 补充后端回归测试，覆盖最新低可信来源无比分、较高可信来源有比分时，舆情总览和详情均输出最高可信来源比分。

## 第一百一十九轮补充：旅行距离、场馆转换与赛程密度

### 问题

### 设计

- 以正式库 `matches.kickoff_time` 和球队参赛顺序为唯一赛程依据，按球队维度寻找上一场比赛。
- 以 ESPN/正式库场馆城市资料加 Open-Meteo Geocoding API 城市中心点坐标建立 16 个场馆映射；距离使用 Haversine 城市中心直线距离，只作为主办城市转换近似值。
- 新增三类因素：
  - `TRAVEL / TRAVEL_DISTANCE`：上一场主办城市到本场主办城市估算距离。
  - `TRAVEL / VENUE_TRANSITION`：同场馆、同城不同场馆、同一国家跨城市、跨国家主办城市转换。
  - `SCHEDULE / SCHEDULE_DENSITY`：滚动 7 天/10 天窗口内比赛场次、距上一场小时数和上一段转场距离。

## 第一百二十八轮补充：关键球员进球参与与贡献分布事实

### 问题

正式库已有 `match_player_stats` 的球员进球与助攻统计，且 48 支球队均有已完赛球员统计记录，但公开页此前只在球员画像中零散展示进球/助攻，缺少球队级“谁参与了进球、贡献是否集中、主要参与者是谁”的事实摘要。若直接称为“核心球员影响”又容易被误读成预计首发、健康状态、未来进球概率或市场方向，因此需要用严格的已完赛统计口径补齐可读事实。

### 设计

- 只从正式库 `match_player_stats + players + teams + matches` 的已完赛球员进球/助攻统计派生，不引入预测、伤停、预计首发、健康、胜率或市场方向。
- 新增 48 条 `team_profile_facts.fact_type=GOAL_CONTRIBUTION`，`period_key=TOURNAMENT_TO_DATE`，展示球队本届球员进球、助攻、进球参与合计、主要参与者和最高贡献者占比。
- 对当前 15 场双方明确且未赛的 32 强赛新增 `ATTACK_CONTRIBUTION / KEY_PLAYER_CONTRIBUTION`，每场主客队各 1 条，共 30 条；按目标比赛 `kickoff_time` 截断，只使用开球前已完赛球员统计。
- 新增 15 条 `source_evidence.source_type=KEY_PLAYER_CONTRIBUTION_DERIVED`，用于追溯每场赛前关键球员进球参与因素的统计来源。
- `source_ref` 统一为 `derived:goal-contribution:team:{team_key}`、`derived:key-player-contribution:match:{match_key}:team:{team_key}`、`derived:key-player-contribution:match:{match_key}`，导入前按同源引用去重。
- 前端补充 `ATTACK_CONTRIBUTION / KEY_PLAYER_CONTRIBUTION / GOAL_CONTRIBUTION / TOP_SCORER_CONTRIBUTION / ASSIST_CONTRIBUTION / CONTRIBUTION_DISTRIBUTION / KEY_PLAYER_CONTRIBUTION_DERIVED` 等中文标签。
- 舆情中心外部因素覆盖矩阵新增“关键球员进球参与”，维度总数改为动态显示；证据中心把关键球员、进攻贡献、进球参与、助攻贡献和贡献分布归入“关键球员进球参与”；球队画像展示“本届进球参与贡献分布”。
- 严格边界：球员进球合计与球队进球可能因乌龙、供应商口径或未归属事件存在差额；贡献占比只基于可归属球员进球+助攻，不等于全部球队进球；不代表预计首发、健康状态、未来进球概率、球员可用性、胜率或市场方向。

## 第一百二十九轮补充：赛前市场价格与隐含概率快照事实

### 问题

正式库已经为当前 15 场双方明确且未赛的 32 强赛保存了 DraftKings `PRE_MATCH` 赔率快照和单点市场信号，但公开舆情/证据页此前仍容易把市场资料表述为“市场价格变化”或“公众热度”。当前数据只有单个赛前时点，没有开盘价、多时间点价格线、公众热度比例或资金比例；若继续展示为变化、热度或流向，会超出数据证据边界。

### 设计

- 只补“赛前市场价格与隐含概率快照”，不新增数据库表，不抓付费源，不补公众热度或资金比例。
- 范围严格限定为当前 `status=SCHEDULED`、`stage=32强赛`、双方球队明确且已有 `PRE_MATCH` DraftKings 快照的 15 场比赛。
- 每场比赛聚合写 1 条正式因素：`match_context_factors.factor_category=MARKET_SIGNAL`、`factor_type=MARKET_PRICE_SNAPSHOT`，摘要包含胜平负、让球/让分、总进球市场的当前价格与隐含概率。
- 每场比赛写 1 条来源证据：`source_evidence.source_type=MARKET_PRICE_SNAPSHOT_DERIVED`，用于证据中心追溯 DraftKings 快照来源。
- `source_ref` 使用 `derived:market-price-snapshot:{match_key}:draftkings:pre-match`，导入前按同源引用去重。
- 前端语义统一：
  - 因素分类显示为“赔率/市场价格”“市场价格信号”。
  - 因素类型显示为“市场价格快照”“市场价格”“隐含概率”“胜平负”“让球/让分”“总进球”。
  - 公开页面静态文案从“市场价格变化”收敛为“市场价格快照”或“价格时间线缺口”。
- 后端公开舆情服务保留动态赔率快照兜底，但当同一比赛已有正式 `MARKET_PRICE_SNAPSHOT` 因素时，抑制从 `odds_market_snapshots` 派生的动态市场卡，避免重复展示。
- 后端公开比赛证据链将 `MARKET_PRICE_SNAPSHOT_DERIVED`、赔率、隐含概率、市场价格相关来源归入“市场 / 赔率信号”。
- 严格边界：不写价格上涨/下跌、盘口变动、赔率异动、公众热度比例、资金流向、市场过热、胜率预测、投注建议或操作指令；单点隐含概率只解释市场价格，不等于球队真实胜率。

## 第一百三十轮补充：门将扑救与防守承压基础画像

### 问题

公开站点已有基础技术风格、进球参与、近期赛果、市场价格和赛制事实，但门将扑救、门将失球、零封和被射正样本仍散落在正式库 `match_player_stats` 与 `match_team_stats` 中。普通用户无法直接看到“门将统计样本是否可读”，也容易把防守动作画像误当作门将扑救质量或真实防线强弱。当前正式库没有 xGOT、PSxG、门将分钟或下一场首发确认，因此必须用更窄的基础统计口径补齐。

### 设计

- 主口径使用 `match_player_stats.payload.stats[*]` 中 GK 球员的 `appearances`、`saves`、`goalsConceded`；不使用当前为空的 `minutes_played`，不换算每 90 分钟。
- 辅助口径使用 `match_team_stats.goals_against` 计算球队失球与零封；对手射正只使用对手 `team_statistics.shotsOnTarget` 的完整 ESPN 技术样本，不使用“对手射正 - 失球”反推扑救。
- 新增球队事实：`team_profile_facts.fact_type=GOALKEEPING_SAVE_PROFILE`，`period_key=TOURNAMENT_TO_DATE`，展示本届门将统计扑救、失球、零封、出场门将和对手射正样本。
- 新增门将球员事实：`player_profile_facts.fact_type=GOALKEEPER_SAVE_PROFILE`，仅给已有正式 GK 出场统计的门将写入，展示出场记录、扑救和门将失球。
- 新增赛前球队侧因素：`match_context_factors.factor_category=GOALKEEPING`、`factor_type=SAVE_PRESSURE_PROFILE`，只针对当前双方明确且未赛的 32 强赛，每场主客队各 1 条，共 30 条。
- 新增来源证据：`source_evidence.source_type=GOALKEEPING_SAVE_PROFILE_DERIVED`，每场 1 条聚合双方来源追溯。
- 去重规则：
  - `derived:team-goalkeeping-save-profile:{team_key}:2026wc`
  - `derived:goalkeeper-save-profile:{player_key}:2026wc`
  - `derived:goalkeeping:save-pressure-profile:{match_key}:{team_fifa_lower}`
  - `derived:goalkeeping-save-profile-evidence:wc2026:{match_key}:through-{yyyymmdd}`
- 前端语义统一：
  - 因素分类显示为“门将/防守承压”。
  - 因素类型显示为“扑救与承压画像”“门将扑救画像”“防守承压画像”。
  - 证据中心覆盖矩阵新增“门将/防守承压”，与“基础技术统计/球队风格画像”区分展示。
- 严格边界：不写 xGOT、PSxG、扑救质量、真实门将能力、防线强弱、伤停健康、训练状态、下一场首发判断、赛果概率、比分倾向或操作方向；不把被射正/扑救数解释为真实压迫强度或战术意图。

## 第一百三十一轮补充：阵容年龄经验与结构基础画像

### 问题

正式库已保存 FIFA 官方 Squad List PDF 的 48 队 26 人名单，包含出生日期、身高、俱乐部、位置、球衣号、赛前国家队出场和进球等基础字段，但公开页此前只展示名单人数和少量球员身份信息。用户无法直接看到球队名单年龄、位置、经验和俱乐部分布结构，也容易把这些名单字段误读成阵容深度、经验优势或临场首发判断。

### 设计

- 只使用正式库 `players.raw_payload.payload` 中的 FIFA 官方 Squad List 字段，不新增外部抓取，不写球员临场可用性。
- 年龄统一按官方名单采集日 `2026-06-24` 计算，避免随当前日期漂移造成同一名单画像前后不一致。
- 新增 48 条球队事实：`team_profile_facts.fact_type=SQUAD_PROFILE`，`period_key=SQUAD_LIST_2026`，展示名单人数、GK/DF/MF/FW 位置分布、平均年龄、U23/30+人数、平均身高、赛前国家队出场/进球合计、50/100 场球员数和俱乐部注册地 Top 分布。
- 不批量写 `player_profile_facts`：1248 名球员基础身份字段可由 `players` 表直接展示，后续如需球员详情增强另开 `PLAYER_SQUAD_IDENTITY` 候选。
- 新增 30 条赛前球队侧因素：`match_context_factors.factor_category=SQUAD_PROFILE`、`factor_type=TEAM_SIDE_ROSTER_BASELINE`，只针对当前 15 场双方明确且未赛的 32 强赛，每场主客队各 1 条。
- 新增 15 条 `source_evidence.source_type=SQUAD_LIST`，每场 1 条聚合双方来源追溯；来源名称统一为 `FIFA 官方 Squad List PDF`，来源 URL 使用正式库保存的 FIFA PDF。
- 去重规则：
  - `derived:team-squad-profile:wc2026:{team_key}:fifa-squad-list-20260624`
  - `derived:match-team-squad-profile:wc2026-r32:{match_key}:{team_key}`
  - `derived:squad-list-evidence:wc2026-r32:{match_key}`
- 前端语义统一：因素分类显示“阵容结构/年龄经验”，因素类型显示“球队名单结构画像”，来源类型显示“官方名单来源”；证据中心覆盖矩阵将其独立于“赛程旅行休息”和“基础技术统计”。
- 严格边界：不写阵容强弱、经验优势、阵容深度强弱、预计首发、伤病恢复、体能状态、未来进球、胜率、比分倾向、投注建议或盘口方向；俱乐部注册地只作名单字段统计，不代表联赛质量或球员能力。

## 第一百三十二轮补充：已确认首发位置结构基础画像

### 问题

正式库已经保存 73 场已完赛的 ESPN 阵容名单，能够确认每队每场 11 名首发及其位置编码，但公开站点此前只在比赛中心和球队页零散列出阵容记录。用户无法直接看到球队本届已确认首发位置结构样本，也容易把“连续首发/轮换压力”等负荷事实误读为阵型或战术。当前正式库没有可靠 `formation` 字段，也没有可用分钟字段，因此不能声称官方阵型、真实战术强弱或体能状态。

### 设计

- 只使用正式库 `match_lineups` 中 `matches.status='FINISHED'` 且 `is_starter=true` 的确认首发样本；目标比赛因素只使用目标开球前已有样本。
- 不使用未赛比赛阵容、不预测下一场名单、不写官方阵型、不写真实战术评价、不写身体状态、胜率、比分倾向或操作方向。
- 新增 48 条球队事实：`team_profile_facts.fact_type=CONFIRMED_STARTING_XI_STRUCTURE`，`period_key=TOURNAMENT_TO_DATE_CONFIRMED_LINEUPS`，展示已入库阵容样本场次、确认首发人次、平均每场首发数、位置编码归并和常见位置编码。
- 新增 30 条赛前球队侧因素：`match_context_factors.factor_category=LINEUP_STRUCTURE`、`factor_type=CONFIRMED_STARTING_XI_STRUCTURE`，只针对当前 15 场双方明确且未赛的 32 强赛，每场主客队各 1 条。
- 新增 15 条 `source_evidence.source_type=LINEUP_STRUCTURE_DERIVED`，每场 1 条聚合双方来源追溯；来源名称说明为正式库 `match_lineups` 与 ESPN `LINEUP` source evidence 派生。
- 不新增球员事实：球员层面首发样本会与既有 `WORKLOAD` 事实重叠，且分钟字段不可用，后续如确需补充只能另开极窄口径 `STARTING_POSITION_SAMPLE`。
- 去重规则：
  - `derived:team-lineup-structure:wc2026:{team_key}:confirmed-lineups-to-20260629`
  - `derived:lineup-structure:wc2026-r32:{match_key}:{team_code_lower}`
  - `derived:lineup-structure-evidence:wc2026-r32:{match_key}`
- 前端语义统一：因素分类显示“首发位置结构”，因素类型和事实类型显示“已确认首发位置结构”，证据来源显示“首发位置结构派生来源”；证据中心将其独立于“阵容结构/年龄经验”和“赛程旅行休息”。
- 严格边界：摘要不得出现“主力稳固”“轮换压力”“预计首发”“首发预测”“战术强弱”“阵型优劣”“疲劳/体能/伤病”“胜率/比分/投注”等判断性或操作性表述。

## 第一百三十三轮补充：犯规越位与点球尝试基础画像

### 问题

正式库 `match_team_stats` 已保存部分 ESPN `team_statistics` 中的 `foulsCommitted`、`offsides`、`penaltyKickShots`、`penaltyKickGoals` 字段，但公开站点此前只在基础技术画像中零散提到犯规，用户无法独立看到犯规、越位和点球尝试样本是否可读。该主题又容易与纪律牌面、裁判判罚和定位球/特殊进球混淆，因此必须以更窄的“基础规则行为样本”口径补齐。

### 设计

- 只使用正式库 `match_team_stats.payload.payload.team_statistics` 中的 ESPN 可用字段：`foulsCommitted`、`offsides`、`penaltyKickShots`、`penaltyKickGoals`。
- 黄牌/红牌字段本轮不纳入画像，避免与 `DISCIPLINE / TEAM_CARD_ACCUMULATION`、球员纪律事实和下场纪律关注重复。
- 新增 48 条球队事实：`team_profile_facts.fact_type=FOUL_OFFSIDE_PENALTY_PROFILE`，`period_key=TOURNAMENT_AVAILABLE_TEAM_STATISTICS`，展示 ESPN team_statistics 可用样本场次、犯规、越位、点球尝试和点球命中。
- 新增 30 条赛前球队侧因素：`match_context_factors.factor_category=INFRACTION_PROFILE`、`factor_type=FOUL_OFFSIDE_PENALTY_PROFILE`，只针对当前双方明确且未赛的 32 强赛，每场主客队各 1 条。
- 新增 15 条来源证据：`source_evidence.source_type=FOUL_OFFSIDE_PENALTY_PROFILE_DERIVED`，每场 1 条聚合双方来源追溯；来源名称为 `正式库 ESPN team_statistics 派生`。
- 不新增球员事实：球员级犯规/越位字段样本会扩展到大量球员，且缺少稳定的球员级点球尝试/命中字段，容易与既有纪律事实重复。
- 去重规则：
  - `derived:foul-offside-penalty-profile:team:{team_key}:2026wc:team-statistics-available-to-20260629`
  - `derived:foul-offside-penalty-profile:wc2026-r32:{match_key}:{team_key}`
  - `derived:foul-offside-penalty-profile-evidence:wc2026-r32:{match_key}`
- 前端语义统一：因素分类显示“犯规/越位样本”，因素类型和事实类型显示“犯规/越位/点球尝试画像”，来源类型显示“犯规/越位样本派生来源”；证据中心将其独立于“基础技术统计/球队风格画像”和“裁判与判罚”。
- 严格边界：不写裁判判罚评价、球队性格评价、纪律价值判断、后续事件预判、赛果推断、胜率、比分倾向、投注建议或操作方向；每条摘要都标明这是 ESPN team_statistics 可用样本，不代表完整球队作风或未来事件。

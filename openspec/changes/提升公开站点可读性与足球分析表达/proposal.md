# 提升公开站点可读性与足球分析表达

## Why

当前公开站点已经有比赛、赔率、舆情、球队/球员画像、证据和赛前作战室，但页面表达仍偏“后台数据仓库”：大量展示条数、缺失、冲突、市场数，普通中国用户进入后很难第一眼理解：

- 这场比赛是谁对谁、有没有完赛、比分是多少、谁赢谁输。
- 哪些数据支持当前判断，哪些数据缺失或过期。
- 球队/球员到底强在哪里、弱在哪里、有没有伤停或状态风险。
- 舆情外因不应只停留在转播平台、场地，还应覆盖天气、裁判、旅行/休息、战意、轮换、伤停、主帅发布会、媒体/球迷情绪、市场热度等。
- PC 端首屏虽然信息很多，但视觉锚点不足，没有国旗、比分、胜负对比、图表和新手解释。

外部参考资料也强调，足球 AI 分析通常需要聚合 xG/xGA/PPDA、伤病、裁判、盘口变动、公众投注、球队新闻、赛程密集、旅行、天气、历史交锋和市场信号等维度（参考：https://www.jenova.ai/zh/resources/best-ai-for-football-betting-analysis ）。本 change 的目标不是生成下注建议，而是把正式库已有/可入库数据转成更直观、更可审查、更适合普通用户理解的公开表达。

## What Changes

### 1. 前端信息架构与视觉表达

- 首页改为“新手一眼看懂”的比赛情报入口：
  - 首屏展示今日/近期焦点比赛卡，包含主客队国旗、球队名、状态、比分/开球时间、完赛胜负、数据准备度、风险摘要。
  - 增加“先看三件事”：比分/赛果、数据占优点、风险与缺失。
  - KPI 从纯条数改为用户语言，例如“有多少场可看”“多少场数据不足”“多少场已完赛有比分”。
- 比赛中心改为 scoreboard + detail：
  - 列表卡直接显示国旗、比分、胜/平/负结果、比赛状态、场地、开球时间。
  - 详情页增加比分牌、进球时间轴、事件时间线、球队关键数据对比条。
- 赛前作战室增加图表化判断框架：
  - 数据准备度 bullet/progress 图。
  - 赔率隐含概率/胜平负对比条（只展示市场数据，不输出下注建议）。
  - 舆情风险分布条、证据可信度分布。
  - 新手解释卡：xG、PPDA、隐含概率、完整度、冲突是什么意思。
- 球队画像中心增强为“球队名片”：
  - 国旗、洲别/小组、近期战绩、进攻/防守画像、进球时间分布、阵容覆盖度。
  - 空字段显示“待补采”，同时提示缺少哪类证据。
- 球员画像中心增强为“可用性与影响”：
  - 位置、号码、所属队、伤停/红黄牌/训练/预计出场状态。
  - 对核心球员突出显示“影响比赛的原因”。
- 舆情与外部因素中心重组分类：
  - 天气/场地、裁判、伤停、赛程密集、旅行距离、战意、轮换、主帅发布会、媒体舆论、球迷热度、市场/公众投注、历史交锋、战术对位。
  - 用风险等级、时效、来源可信度和建议核查动作呈现。
- 证据中心增强为“证据质量面板”：
  - 每条证据显示来源类型、可信度、更新时间、是否过期、支撑哪个结论。
  - 冲突项明确“哪里冲突、需要人工核对什么”。

### 2. 后端公开 DTO 与聚合能力

- 在 public API 中增加面向前端可视化的派生字段：
  - `homeScore`、`awayScore`、`scoreDisplay`、`winnerSide`、`resultText`。
  - 主客队 `fifaCode`、`countryIso2`、`flagUrl` 或可本地映射的 flag key。
  - `matchVisualSummary`：比赛状态、比分、完赛结论、比分来源、数据准备度。
  - `teamComparison`：进球、失球、首球时间、事件数、赔率隐含概率、证据数等。
  - `glossaryHints`：页面涉及的术语解释 key。
- 保持公开 API 脱敏边界：不暴露票号、资金、下注金额建议、原始 JSON 敏感字段。
- 对现有 `match_team_stats`、`odds_selection_snapshots`、`source_evidence` 做只读聚合，优先从正式库派生比分和结果，不在前端硬算业务结论。

### 3. 数据库与数据健壮性

- 为球队补充或规范国家元数据：`country_iso2`、`flag_asset_key`、洲别、小组信息来源追踪。
- 评估新增结构化指标快照表：
  - `team_metric_snapshots`：xG、xGA、npxG、PPDA、射门、控球、预期积分、近期状态等。
  - `player_metric_snapshots`：分钟、进球、助攻、xG、xA、射门、关键传球、伤停/训练状态来源等。
  - `match_market_signals`：盘口变化、隐含概率、公众投注百分比、资金百分比、变化时间线。
- 若本阶段先不新增表，则先复用 `team_profile_facts`、`player_profile_facts`、`match_context_factors` 和 `source_evidence`，但前端必须明确“待补采/待入库”。

### 4. 数据导入与审核材料

- 扩展 SOURCE/TEAM/PLAYER/MATCH JSON 支持的结构化字段，但仍遵循现有数据流：新 JSON 进入 `data-inbox/pending/`，经审核和正式入库后才成为权威数据。
- 提供示例字段模板，方便后续补充：伤停、裁判、天气、旅行、赛程密集、战意、市场热度、xG 指标等。

## Out of Scope

- 不生成“推荐下注”“必中”“稳赚”“加注/倍投”等投注指令。
- 不绕过 JSON 审核入库流程，不把 pending JSON 当权威数据。
- 不自动抓取付费数据源，不保存新的外部密钥。
- 不修改 `CLAUDE.md` 与已移除的旧 `skill/` 流程。
- 不承诺预测准确率；所有图表只解释数据、来源、风险和完整度。

## Review Gate

实施前：

1. 用户确认本 OpenSpec change 可进入实施。
2. 明确第一轮优先级：建议先做“比赛/首页/作战室可视化”，再做数据库指标扩展。

交付前：

1. `mvn -f server/pom.xml test` 通过。
2. `npm run test:run` 在 `client/` 通过。
3. `npm run build` 在 `client/` 通过。
4. `git diff --check` 通过。
5. 浏览器 PC 视口截图回归：首页、作战室、比赛中心、舆情、球队、球员至少各一张。
6. 公开页检查：无资金/票号/原始 JSON 敏感字段泄露。
7. 新手验收：首屏无需懂足球也能看懂“谁对谁、比分/时间、当前状态、主要风险、数据是否足够”。

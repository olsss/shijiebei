# 核心数据模型设计

日期：2026-06-22（北京时间）
阶段：第三阶段
OpenSpec change：`核心数据模型`
状态：用户已确认阶段方向，待实现计划

## 1. 背景

前两阶段已经完成 Java 前后端骨架和 JSON 审核批准入库流程。当前系统可以扫描 `skill/archive/` 下的 `bets.json`、`analysis/*.json`、`odds/*.json`、`sources/*.json`，把 JSON 保存为 `import_jobs` / `import_items`，并支持批准、批量批准、驳回和审计。

第三阶段的目标是建立正式 MySQL 业务库地基，让已批准 JSON 可以进入结构化业务表。该阶段仍然不替代 `CLAUDE.md` 与 `skill/` 的比赛分析规则，不生成比赛结论，不自动给下注建议。

## 2. 真实 JSON 结构依据

已抽样检查当前档案：

- `skill/archive/bets.json`
  - 顶层字段：`edition`、`currency`、`note`、`schema`、`bets`
  - `bets[]` 包含下注日期、比赛日、类型、比赛、让球数、下注内容、投入、赔率、总赔率、CLV、赛果、命中、返还、盈亏、决策方、证据状态、结论类型、备注等。
- `skill/archive/analysis/*.json`
  - 常见字段：`id`、`status`、`date_analyzed`、`matchday`、`match`、`jc_code`、`odds_file`、`odds_ref`、`value_flags`、`dimensions`、`temperament`、`per_match_form`、`conclusion_type`、`confidence`、`risks`、`recommended`、`bet_ids`、`narrative_md`、`result`、`sources`。
- `skill/archive/odds/*.json`
  - 常见字段：`match`、`date`、`commence_utc`、`jc_code`、`source`、`books_count`、`official`、`companies`、`all_books`、`markets`、`date_beijing`、`commence_beijing`、`timezone`。
- `skill/archive/sources/*.json`
  - 常见字段：`match`、`date`、`snapshots`、`tier_priority`、`field_coverage`、`conflicts`、`status`、`note`、`aliases`、`last_updated`、`jc_code`、`fifa_match_id`、`competition`。

本阶段数据模型必须兼容这些字段，同时允许部分字段缺失。结构化字段用于查询、统计和关联；原始 JSON 仍以 `import_items.raw_json` 为追溯源。

## 4. 数据建模原则

1. **批准优先**：只有 `import_items.status = APPROVED` 且 `valid_json = true` 的条目可进入正式表。
2. **幂等导入**：同一 `import_item_id` 只能导入一次。结果记录在 `import_item_mappings`。
3. **原文可追溯**：正式表保存 `import_item_id` 或证据引用，不复制替代原始 JSON。
4. **弱结构兼容**：真实 JSON 字段可能不完整，核心字段结构化，复杂对象可先用 JSON/LONGTEXT 存储。
5. **证据链优先**：来源、冲突、可信度、更新时间必须能落表。
6. **标准化逐步增强**：球队/球员/玩法字典先建立基础表和 alias 机制，复杂去重和冲突解决后续增强。

## 6. 后端设计

新增包：`com.worldcup.coredata`。

建议结构：

```text
coredata/
├── api/
│   ├── CoreDataController.java
│   └── dto/
├── domain/
├── repo/
├── service/
│   ├── CoreDataImportService.java
│   ├── BusinessJsonMapper.java
│   ├── MatchKeyNormalizer.java
│   └── CoreDataOverviewService.java
└── support/
```

核心服务：

- `CoreDataImportService`
  - 校验 import item 是否存在、是否 approved、是否 valid。
  - 调用对应 mapper。
  - 写入 `import_item_mappings` 和 `audit_logs`。
- `BusinessJsonMapper`
  - 根据 `ImportItemType` 分发到 analysis / odds / sources / bets 映射逻辑。
- `CoreDataOverviewService`
  - 汇总正式业务表数量与最近入库状态。
- `MatchKeyNormalizer`
  - 将 `match`、`matchday`、`jc_code` 标准化为 `match_key`。

## 7. 映射规则

### 7.1 ANALYSIS

输入：`analysis/*.json`。

输出：

- `matches`：按 `match + matchday + jc_code` upsert。
- `analysis_reports`：保存分析摘要、维度 JSON、来源 JSON、长文。
- `source_evidence`：从 `sources[]` 提取来源摘要。
- `data_conflicts`：如果 JSON 中有冲突字段，保存冲突记录。

### 7.2 ODDS

输入：`odds/*.json`。

输出：

- `matches`：按 match/date/jc_code upsert。
- `odds_snapshots`：展开 `companies`、`markets` 或 `all_books` 中可识别赔率；无法结构化的部分放入 `raw_payload`。

### 7.3 SOURCE

输入：`sources/*.json`。

输出：

- `matches`：按 match/date/jc_code upsert。
- `source_evidence`：展开 `snapshots`。
- `data_conflicts`：展开 `conflicts`。
- `data_dictionaries`：可从 `aliases` 建立球队/名称别名。

### 7.4 BETS

输入：`bets.json`。

输出：

- `bets`：逐条保存 `bets[]`。
- `matches`：按 `比赛 + 比赛日` upsert。
- `analysis_reports` 不由 bets 生成，仅通过 `bet_ids` 关联在后续阶段增强。

## 8. API 设计

所有接口继续使用 Basic Auth 管理员认证。

```text
GET  /api/core-data/overview
POST /api/core-data/import-items/{itemId}/import
GET  /api/core-data/import-items/{itemId}/mappings
```

响应统一使用 `ApiResponse.ok(...)`。

导入错误：

- import item 不存在：404。
- 未批准或 JSON 无效：400。
- 已导入：返回现有 mapping 或 409，实施阶段优先选择幂等返回现有 mapping。
- JSON 字段缺失但可部分入库：保存可识别字段，message 写明降级。

# Java 世界杯竞彩个人管理系统总体设计

日期：2026-06-22（北京时间）
状态：已由用户口头确认，待用户 review 本文件

## 1. 背景与目标

当前项目已经具备一套围绕世界杯竞彩分析的本地规则、JSON 档案、脚本和看板：

- 根目录 `CLAUDE.md` 是比赛分析与下注记录的权威规则。
- `skill/SKILL.md` 与 `skill/rules/` 管理比赛分析流程。
- `skill/archive/bets.json`、`skill/archive/analysis/*.json`、`skill/archive/sources/`、`skill/archive/odds/` 保存 AI/Claude/Codex 产出的分析、下注、来源与赔率档案。

本设计目标是把现有项目扩展为一个 Java 系统项目，分为前端客户端和后端管理数据端，同时保留原有比赛分析体系。

系统定位：

> 个人使用的完整世界杯竞彩数据分析管理工作台。

核心原则：

1. AI/Claude/Codex 继续负责比赛分析、证据采集和 JSON 档案生成。
2. 用户人工审核批准后，JSON 数据才进入 MySQL 正式库。
3. Java 系统负责管理、审核、入库、展示、统计、追溯和复盘。
4. OpenSpec 只管理 Java 系统开发流程，不参与比赛分析与下注判断。
5. UI/UX Pro Max skill 只用于 Java 系统客户端界面重写，不参与比赛分析。

## 4. 数据权威与审批策略

### 4.1 权威关系

```text
AI/Claude/Codex 分析比赛
    ↓
生成 JSON 草稿与证据档案
    ↓
用户人工审核
    ↓
后台页面批准 或 CLI 命令批准
    ↓
写入 MySQL 正式库
    ↓
前端客户端展示、查询、统计、复盘
```

### 4.2 数据源定位

- JSON：AI/Claude/Codex 产出的原始分析档案、证据档案、待审核数据。
- MySQL：用户批准后的正式业务库。
- Java 后台：审核、预览、批准入库、回滚、版本追踪。
- 前端客户端：查询、展示、统计、审核操作入口。

### 4.3 审核状态流

```text
draft_json
  ↓
pending_review
  ↓
approved / rejected
  ↓
imported_to_mysql
  ↓
archived
```

要求：

- 未批准 JSON 不进入正式看板。
- 已入库数据发现错误时，支持撤回、重新导入、保留原版本、记录修改原因。
- 所有批准、驳回、导入、撤回动作写入审计日志。

## 6. 数据导入审核模块

第一版核心模块。

功能：

- 扫描 `skill/archive/bets.json`
- 扫描 `skill/archive/analysis/*.json`
- 扫描 `skill/archive/sources/*`
- 扫描 `skill/archive/odds/*`
- 生成导入任务 `import_job`
- 拆分导入条目 `import_item`
- 字段完整性校验
- 数据字典匹配校验
- 来源证据校验
- 显示待审核、已批准、已驳回、已入库状态
- 支持查看 JSON 原文
- 支持结构化预览
- 支持单条批准入库
- 支持批量批准入库
- 支持驳回并填写原因
- 支持入库日志查看
- 提供 CLI 备用导入命令

CLI 预期形式：

```bash
java -jar server.jar import-json --path skill/archive --dry-run
java -jar server.jar import-json --path skill/archive --approve
```

`--dry-run` 只校验不入库；`--approve` 才执行批准导入。

## 7. 球队画像中心

球队不是简单名称表，必须单独建模。

### 7.1 球队基础档案

字段包括：

- 球队名称
- 英文名
- 所属大洲 / 足协
- 小组
- FIFA 排名
- 主教练
- 常用阵型
- 核心球员
- 平均年龄
- 平均身高
- 世界杯 / 洲际杯 / 预选赛履历
- 本届世界杯状态标签

### 7.2 球队历史与本届表现

记录：

- 对手
- 比分
- 比赛阶段
- 进球时间点
- 失球时间点
- 控球率
- 射门 / 射正
- 角球
- xG / 机会质量（如果有）
- 是否先进球
- 领先后是否守得住
- 落后后是否急躁
- 是否擅长下半场发力
- 是否容易最后阶段丢球

### 7.3 进球时间点分析

时间段：

```text
0-15 分钟：开局强度
16-30 分钟：压迫稳定性
31-45+ 分钟：半场前风险
46-60 分钟：中场调整效果
61-75 分钟：体能与换人
76-90+ 分钟：守领先 / 绝杀 / 崩盘风险
```

可生成标签：

- 慢热型
- 开局猛攻型
- 下半场发力型
- 领先保守型
- 最后阶段易失球
- 逆风球能力弱
- 定位球得分强
- 反击效率高

### 7.4 球队特性标签

标签包括：

- 低位防守能力
- 反击能力
- 定位球能力
- 身体对抗
- 破密集能力
- 守领先能力
- 控球是否有效
- 大赛稳定性
- 必须赢时是否急躁
- 面对弱队是否容易卡盘
- 面对强队是否有韧性

每个标签必须绑定证据：

```text
标签：破密集能力弱
证据：近 3 场面对低位防守射正少、传中多但机会质量低
来源：比赛数据 + 战报 + 人工分析
置信度：中
更新时间：2026-06-22
```

- 是否有纪律风险

### 8.4 球员每场表现

记录：

- 是否首发
- 出场时间
- 被换上 / 被换下时间
- 进球
- 助攻
- 射门 / 射正
- 关键传球
- 过人
- 抢断 / 拦截 / 解围
- 犯规 / 被犯规
- 黄牌 / 红牌

- xG / xA（如果有）
- 评分（如果有）
- 战术职责
- 表现评价

### 8.5 球员更衣室与舆情

记录：

- 队内矛盾传闻
- 与主教练关系
- 与队友关系
- 续约 / 转会问题
- 社媒争议
- 发布会态度
- 媒体批评
- 球迷压力
- 是否可能影响出场意愿或专注度

来源分级：

```text
官方确认 > 主流记者 > 主流媒体 > 普通媒体 > 社媒传闻
```

每条记录保存：

- 内容
- 来源
- 发布时间
- 可信度
- 影响判断
- 是否已被证实 / 否认 / 待确认

### 8.6 球员标签

标签包括：

- 终结能力强
- 机会浪费多
- 速度爆点
- 高空优势
- 定位球威胁
- 防守冒失
- 易吃牌

- 大赛经验足
- 替补冲击力强

## 15. 数据来源、证据链与冲突处理

### 15.1 证据链字段

每条球队、球员、比赛、舆情、赔率数据都要记录：

- 来源名称
- 来源链接/文件
- 抓取时间
- 可信等级
- 是否官方确认
- 是否与其他来源冲突
- 使用该数据的分析报告

### 15.2 数据版本

对变化频繁的数据保留版本历史：

```text
球员 A：
2026-06-20：疑似伤病
2026-06-21：恢复合练
2026-06-22：确认首发
```

### 15.3 冲突数据处理

冲突不能直接覆盖，应保存冲突记录：

```text
来源 A：球员出战成疑
来源 B：球员已参加合练
处理结果：标记为部分可信，赛前再核
```

## 16. 数据字典与标准化

必须标准化：

- 球队名：巴西 / Brazil / BRA 指向同一球队
- 球员名：中文名、英文名、别名统一映射
- 位置：中锋 / CF / ST 映射到标准位置
- 玩法：HAD / 胜平负；HHAD / 让球胜平负
- 比赛阶段：小组赛、淘汰赛等
- 来源等级
- 置信度等级
- 审核状态

## 17. 数据时效提醒

系统要提醒过期数据：

- 伤病 / 首发 / 停赛 / 临场消息超过 24 小时
- 赔率 live 距停售超过 3 小时
- 球队档案或小组排名未含最新一轮或超过 7 天
- 舆情来源超过一定时间未确认

过期数据不自动作废，但必须触发提醒、降级或要求补采。

## 18. 备份与导出

个人系统也必须防止数据丢失：

- MySQL 定期导出
- JSON 原档案不删除
- 支持 MySQL 数据重新导出 JSON
- 支持导出分析报告 Markdown / HTML
- 支持导出复盘报告

## 26. 风险与约束

1. 范围很大，必须通过 OpenSpec 分 change 推进。
2. JSON 审核入库是系统成败关键，不能直接跳到复杂 UI。
3. 球队、球员、舆情、赔率都是高变化数据，必须保留版本与时效提醒。
4. 舆情只能作为观点和风险提示，不能当作事实。
5. MySQL 是已批准正式库，不替代 JSON 原始档案。
6. Java 系统不自动做比赛判断，避免与 `CLAUDE.md` / `skill/SKILL.md` 的分析流程冲突。
7. UI/UX Pro Max skill 当前未在可用列表中，实施前需要安装或提供路径。

# 球队球员采集画像设计

## 背景

当前系统已经完成 JSON 审核入库与核心数据模型：AI/Claude/Codex 生成的数据先进入 `import_items`，批准后再映射进正式 MySQL 业务表。本阶段在此基础上完成球队与球员画像模块，让本人可以查看球队特性、上阵人员、首发阵容、历史进球时间点、外部因素、舆情，以及球员状态、伤病、红黄牌和更衣室情况。

本阶段遵守边界：Java 系统负责数据管理、证据链、冲突提示和画像展示，不自动给出下注结论，不修改 `CLAUDE.md` 或 `skill/` 规则。若后续 Java 结构化事实与 skill 分析流程冲突，单独发起 `skill规则同步` change。

## 目标

1. 建立球队/球员画像事实表与采集暂存表。
2. 允许外部/AI 采集结果先进入待审核区，批准后写入正式画像事实。
3. 后端提供球队列表、球队详情、球员列表、球员详情、采集项审核 API。
4. 前端提供球队画像中心、球员画像中心、采集审核入口。
5. 画像中必须展示来源、更新时间、可信度、冲突状态，避免把未经批准的数据当事实。

## 数据模型

新增 Flyway `V4__create_profile_tables.sql`：

- `collection_jobs`：采集任务，记录来源类型、关键词、状态、触发人和统计。
- `collection_items`：采集暂存项，保存 entity 类型、entity key、事实类型、摘要、原始 JSON、审核状态。
- `team_profile_facts`：球队画像事实，按 team、fact_type、period_key 记录战术、进球时间、外部因素、舆情等。
- `player_profile_facts`：球员画像事实，按 player、fact_type、period_key 记录状态、伤病、红黄牌、更衣室、表现等。

事实类型采用字符串枚举而不是硬编码表结构：`STYLE`、`LINEUP`、`SCORING_PATTERN`、`HISTORY`、`EXTERNAL_FACTOR`、`SENTIMENT`、`FORM`、`INJURY`、`CARD`、`LOCKER_ROOM`、`PERFORMANCE`。这样能覆盖方案 C 的舆情与外部因素，同时不提前引入复杂爬虫/模型服务。

## 后端设计

新增包 `com.worldcup.profile`：

- `ProfileCollectionService`：创建人工采集任务、写入采集暂存项、批准/驳回采集项。批准时只写画像事实表，不直接覆盖 `teams` 或 `players` 基础字段。
- `ProfileQueryService`：聚合球队/球员基础表、画像事实、阵容、比赛统计、证据链和冲突数量。
- `ProfileController`：暴露 `/api/profiles/**` 管理接口。

关键 API：

- `POST /api/profiles/collections/jobs`
- `GET /api/profiles/collections/items?status=PENDING_REVIEW`
- `POST /api/profiles/collections/items/{id}/approve`
- `POST /api/profiles/collections/items/{id}/reject`
- `GET /api/profiles/teams`
- `GET /api/profiles/teams/{teamId}`
- `GET /api/profiles/teams/{teamId}/players`
- `GET /api/profiles/players`
- `GET /api/profiles/players/{playerId}`

所有接口继续使用管理员认证。

## 错误处理与安全边界

- 采集项必须存在且为 `PENDING_REVIEW` 才能批准或驳回。
- 批准时如果 entity 不存在，返回 400，不自动创建模糊球队/球员。
- 事实保留 `source_name`、`source_url`、`source_ref`、`reliability_score`、`captured_at`、`approved_by`。
- 同一采集项重复批准保持幂等，不重复写事实。
- 不写 `skill/archive/`，不修改 `CLAUDE.md`，不生成下注建议。

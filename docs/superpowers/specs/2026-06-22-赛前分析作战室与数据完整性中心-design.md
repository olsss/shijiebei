# 赛前分析作战室与数据完整性中心设计

日期：2026-06-22（北京时间）
状态：方案 A 已确认，进入 OpenSpec 文档阶段

## 背景

项目已完成 JSON 审核入库、核心数据模型、球队/球员画像、比赛中心、赔率中心、舆情与外部因素中心、分析下注复盘中心。当前缺口是：单场比赛赛前检查时，数据分散在多个中心，无法一眼判断这场比赛是否已经具备足够的数据、是否存在过期数据和未解决冲突。

## 目标

构建个人使用的“赛前分析作战室 + 数据完整性中心”：

- 以比赛为入口聚合所有正式业务数据。
- 展示球队、球员、阵容、赔率、舆情、证据、分析报告、AI 下注方案和实际下注记录。
- 计算完整性分数、缺失项、过期项和冲突项。
- 前端提供 `/prematch-workbench` 页面和 Dashboard 入口。
- 保持 Java 系统边界：只聚合、校验、展示已批准入库数据，不生成新分析或下注建议。

## 非目标

- 不新增采集器或外部接口。
- 不新增数据库表。
- 不生成新的下注方式、投注方向、投注金额、加注、倍投或保证命中文案。
- 不修改 `CLAUDE.md`、`skill/` 规则与档案。

## 数据流

```text
已批准 JSON 入正式业务表
  ↓
PrematchWorkbenchQueryService 按 match_id 聚合
  ↓
完整性规则计算 PASS/MISSING/STALE/CONFLICT
  ↓
/api/prematch-workbench/*
  ↓
/prematch-workbench 页面集中展示
```

## 完整性规则

共 10 个检查项：

1. 主客队球队画像。
2. 阵容球员画像。
3. 阵容与首发。
4. 赔率市场快照。
5. live 赔率 3 小时时效。
6. 舆情与外部因素时效。
7. 分析报告。
8. AI 下注方案。
9. 至少 2 条来源证据。
10. 未解决数据冲突。

完整性分数 = `PASS` 数量 / 总检查数量 * 100。分数只表示数据准备程度，不表示投注胜率或下注价值。

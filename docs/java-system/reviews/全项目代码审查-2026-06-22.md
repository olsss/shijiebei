- 审查时间：2026-06-22（Asia/Shanghai）
- 审查范围：Java 后端、Vue 前端、Flyway 数据库迁移、JSON 审核/正式入库链路、OpenSpec 记录、测试与构建验证
- 审查原则：本次只做代码审查与问题记录，未做 MySQL 联调，未修改 `CLAUDE.md`、`skill/`、`skill/rules/`、`skill/archive/`

## 1. 总结结论

当前项目已经具备前端客户端 + 后端管理数据端的完整模块骨架：JSON 审核中心、核心数据概览、比赛中心、球队/球员画像、赔率中心、舆情中心、分析下注复盘中心、赛前作战室均已存在，并且前端构建通过。

但从“可以长期作为本人真实使用的数据系统”的角度看，**还不能判定为全部完成**。最大缺口在正式数据入库链路：现在批准后的 JSON 主要能导入分析、赔率、来源证据、舆情、下注方案和下注记录；**球队、球员、主客队关联、首发阵容、比赛事件、球队技术统计、球员技术统计等基础事实表没有对应的自动 JSON 入库 mapper**。这会导致赛前作战室、球队画像和球员画像页面有表、有 UI、有查询，但真实数据只能依赖手动 SQL/测试夹具或后续补充。

另外，后端全量测试在本次审查中出现过 1 个 flaky 失败，说明测试稳定性需要先修。

## 3. 高优先级问题

### H1. 正式入库链路缺少球队/球员/阵容/事件/技术统计 mapper

证据：
- `JsonArchiveScanner` 当前只扫描 `bets.json`、`analysis/`、`odds/`、`sources/`。
- `BusinessJsonMapper` 只分发 `ANALYSIS`、`ODDS`、`SOURCE`、`BETS`。
- 生产代码中未发现对 `teams`、`players`、`match_lineups`、`match_events`、`match_team_stats`、`match_player_stats` 的正式导入写入；这些表主要由测试夹具直接插入。

影响：
- 你要求的“每只球队的上阵人员、首发阵容、球队历史数据、进球时间点、外部因素、舆情分析、球员状态、伤病、红黄牌、更衣室情况”等，目前只有部分表结构和展示能力，没有完整 JSON → MySQL 自动入库链路。
- 赛前作战室完整性检查会长期显示缺球队画像/球员画像/阵容，除非后续人工录入或补 mapper。

建议下一阶段优先新建 OpenSpec change：`基础事实数据入库与实体关联`。

### H2. `upsertMatch` 只创建最小比赛记录，未关联主客队

证据：`BusinessJsonMapper.upsertMatch(...)` 只写入 `match_key`、`match_name`、`matchday`、`jc_code`、`status`、`result_status`、`raw_payload`，没有解析/写入 `home_team_id`、`away_team_id`、`competition`、`stage`、`venue`、`kickoff_time`。

影响：
- 即便导入了分析、赔率、下注记录，也可能只得到“法国 vs 巴西”这样的孤立比赛；球队画像、阵容、历史表现无法可靠挂到该比赛。

### H3. 后端全量测试存在 flaky 失败

失败现象：
- 全量 `mvn test` 曾失败：断言 `$.data.evidence[0].sourceName` 期望 `FIFA`，实际返回 `Opta`。

根因：
- 测试夹具连续两次用 `LocalDateTime.now().minusHours(1)` 插入证据；第二条时间戳可能更晚。
- 服务按 `evidence_time DESC` 排序，所以第一条不稳定。

建议：
- 固定测试时间戳，或改为断言集合包含 `FIFA`/`Opta`，或明确统一证据排序规则。

### H4. 本地单人使用可以接受，但如果对外暴露存在安全边界问题

证据：
- `application.yml` 默认 MySQL 为 `root/root`，管理员为 `admin/admin123456`。
- 登录页默认填充 `admin/admin123456`。
- `POST /api/import-jobs/scan` 接收任意 `archivePath`，后端会按该路径读取 JSON。

影响：
- 个人本机使用可以，但一旦局域网/公网暴露，任意已登录用户可扫描进程可读路径，默认密码也不安全。

建议：
- 继续保留本地默认也可以，但启动脚本/文档应强制提示生产或长期使用必须设置环境变量。
- 扫描路径建议限制在配置的 `WORLDCUP_ARCHIVE_PATH` 根目录内。

## 4. 中优先级问题

### M1. 缺少统一异常处理

目前成功响应统一为 `ApiResponse`，但 `ResponseStatusException`、校验异常、空请求等错误没有统一 `@RestControllerAdvice` 包装。前端捕获到的多是 Axios/Spring 默认错误，用户提示不够稳定。

### M2. 输入校验不完整

已有 `LoginRequest` 使用 `@NotBlank`，但 `BatchApproveRequest`、`RejectImportItemRequest`、`CreateCollectionJobRequest` 等多数 DTO 没有 Bean Validation。部分 null body 或字段异常会走手工校验，部分可能产生 500 或不一致错误。

### M3. 数据模型还缺少若干“真实分析必需”的结构化字段

当前可通过 `raw_payload` 暂存，但不利于查询/排序/完整性检查的内容包括：
- 小组积分/排名/最佳第三名横向对比；
- 教练、战术体系、轮换倾向；
- 伤病/停赛/红黄牌历史时间线；
- 竞彩官方玩法开售状态、让球数、停售时间、票面 SP；
- 比赛最终比分字段；
- 冲突解决记录与人工批准后的 resolution 过程。

### M4. `data_conflicts.match_id` 缺少索引

`data_conflicts` 有 `match_id` 外键，但只建了 `idx_data_conflicts_import_item`。比赛中心和赛前作战室会按 `match_id` 查询冲突，后续数据多了建议补 `idx_data_conflicts_match`。

### M5. 时间解析会忽略时区偏移

`BusinessJsonMapper.parseDateTime` 会去掉 `Z` 并截取前 19 位；类似 `2026-06-22T10:00:00+08:00` 的偏移会被静默忽略。赔率采集、停售时间、外部来源时间建议使用 `OffsetDateTime`/`Instant` 标准化。

### M6. 核心数据概览统计不完整

Dashboard 只统计 `teams`、`players`、`matches`、`analysis_reports`、`bets`、`odds_snapshots`、`source_evidence`、`import_item_mappings`。新增模块表如 `odds_market_snapshots`、`odds_selection_snapshots`、`bet_plans`、`match_context_factors`、画像 facts、冲突数等没有进入概览。

### M7. 画像采集 `captured_at` 缺少输入与默认写入

`CreateCollectionJobRequest` 没有 `capturedAt` 字段，`ProfileCollectionService.createJob` 写入 `collection_items` 时也没有写 `captured_at`。批准后的画像事实可能缺少采集时间，影响时效判断。

## 5. 前端问题

### F1. 无全局路由守卫

所有业务路由都可直接打开，页面内部再因没有 auth header 报错。建议加 `router.beforeEach`，未登录统一跳 `/login`。

### F2. 无 Axios 全局错误拦截

每个页面手写 catch，401/403/500 体验不统一。建议在 `http.ts` 增加响应拦截器，统一处理未登录、服务不可达和后端错误 message。

### F3. 构建包过大

Vite 构建通过，但主 JS chunk 约 1.16MB。建议后续对各业务页面做路由懒加载，并对 Element Plus/ECharts 等做分包。

### F4. 前端测试偏 API helper，UI 行为覆盖少

现有前端 14 个测试通过，但多数是 API path/helper/router 测试。建议补充登录跳转、未登录拦截、导入按钮状态、赛前作战室展开行等关键 UI 流程测试。

## 6. 其他注意事项

- `.env` 中存在本地 Odds API key，当前 `.gitignore` 已忽略，未被 git 跟踪；如果该项目目录会共享/截图/打包，建议轮换 key。
- OpenSpec 当前 9 个 change 都能 validate，但 `openspec list --specs` 显示暂无 specs。后续长期维护时建议把已完成能力沉淀为 specs 基线，避免只有 change 历史没有当前规范。
- `admin_users`、`system_settings` 表已建，但当前实际认证仍使用配置文件内存用户，系统设置也只读配置；后续如果要真正“后端管理数据端”，可以补管理界面和数据库化配置。

## 7. 建议下一阶段

建议下一阶段不要先做联调，而是先补一个中文 OpenSpec change：

**`基础事实数据入库与实体关联`**

建议范围：
1. 扩展 JSON 扫描类型/目录：`teams/`、`players/`、`matches/`、`lineups/`、`events/`、`stats/`，或在 `sources/*.json` 中支持结构化数组。
2. 新增 mapper：
   - upsert team；
   - upsert player；
   - upsert match 并关联主客队；
   - import lineups；
   - import match events；

3. 补测试：
   - JSON 导入后球队/球员/比赛能关联；
   - 赛前作战室完整性从缺失变通过；
   - 时间戳固定，移除 flaky；
   - malformed request 返回统一错误。
4. 顺手补小修：
   - `data_conflicts(match_id)` 索引；
   - 全局异常处理；
   - 路由守卫和 Axios 拦截器。

## 8. 审查结论

模块“页面和接口”层面基本齐了；但“真实可持续使用的数据链路”还没闭环。下一步最重要的是把球队、球员、比赛事实、阵容、事件、统计这些基础事实数据正式导入并与现有分析/赔率/舆情/下注表关联起来。

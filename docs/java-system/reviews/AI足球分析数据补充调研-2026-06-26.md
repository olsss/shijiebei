# AI 足球分析数据补充调研与当前项目缺口

- 调研时间：2026-06-26（Asia/Shanghai）
- 调研目的：在用户提供 Jenova 文章基础上，补充更可靠/更工程化的足球 AI 分析资料，并对当前正式库、数据入口和公开前端提出数据补充与修改方向。
- 项目边界：本系统以数据库为长期权威源；JSON 只作为临时投递与审核材料。公开前端只做数据解释、风险核查与资料准备度展示，不输出操作指令或确定性预测。

## 1. 参考资料要点

### 1.1 用户提供文章：Jenova 足球 AI 分析文章

链接：https://www.jenova.ai/zh/resources/best-ai-for-football-betting-analysis

可取之处：它给出了一套实用的数据聚合清单：xG/xGA/npxG/PPDA/渐进传球、伤病、裁判倾向、市场价格变化、公众热度、球队新闻、赛程密集、旅行、天气、历史交锋与战术对位。

不足：文章偏产品/营销视角，部分准确率叙述需要谨慎对待，不能直接作为本项目的预测承诺。更适合用作“数据维度清单”，不适合作为模型效果证明。

### 1.2 Opta / Stats Perform：xG、xA 和事件定义

- Opta xG 解释：https://theanalyst.com/articles/what-is-expected-goals-xg
- Stats Perform 事件定义：https://www.statsperform.com/opta-event-definitions/

关键启发：

- xG 是射门质量指标，不等于单场“应该赢”。它适合做球队进攻质量、机会质量和回归候选的基础信号。
- xG 的输入通常包括射门距离、角度、受压程度、身体部位、进攻方式、助攻类型、门将/防守者位置等，因此只保存最终比分远远不够。
- xA 可用于刻画创造机会能力，特别适合球员画像和关键影响面板。
- 不同供应商的 xG 口径并非完全一致，正式库必须保存 `source_name/source_ref/captured_at/metric_type/period_key`，避免混用。

### 1.3 StatsBomb Open Data：事件、阵容和比赛元数据结构

链接：https://github.com/statsbomb/open-data

关键启发：

- 专业足球数据至少分为 competition/season、matches、events、lineups；更高阶会有 360/位置数据。
- 当前项目已有 `matches`、`match_events`、`match_lineups`、`match_team_stats`、`match_player_stats`，方向正确；但若要支撑 AI 分析，需要更细的射门/传球/压迫/定位球/转换进攻等事件指标。
- 可先用 `team_metric_snapshots` 与 `player_metric_snapshots` 承接聚合指标；若后续要保存逐射门 xG、freeze-frame、压迫事件，应再走 OpenSpec 新增结构化事件扩展。

### 1.4 机器学习足球赛果预测综述：Bunker / Yeung / Fujii

链接：https://arxiv.org/pdf/2403.07669

关键启发：

- 足球低比分、多平局，预测天然困难；不能用“高胜率神话”做公开承诺。
- 文献把模型大致分成统计模型、机器学习/图模型、评级系统；Poisson、Dixon-Coles、Elo/pi-ratings、XGBoost/CatBoost 等都有使用场景。
- 更重要的是特征工程和领域知识：球队攻防、近期表现、主客场、对手强度、红黄牌、犯规、控球、传球、跑动、球员年龄/身体特征/可用性、伤病、战意/心理等。
- 模型输出应是概率分布，并用 RPS、Brier、log loss、校准曲线等评价；本项目前端不应展示“必中/稳赢”式结论。

### 1.5 人类专家/媒体预览 + 机器学习：Beal 等

链接：https://arxiv.org/pdf/2012.04380

关键启发：

- 只用基础统计可能遇到效果上限；媒体预览、专家文本和外部背景能补充“心态、德比、竞争关系、战意”等人类因素。
- 对本项目来说，`match_context_factors` 不应只存场地/转播/裁判，还应结构化沉淀发布会、媒体/球迷舆情、战意、轮换、更衣室等文本线索，并记录来源可靠性与时效。

### 1.6 Sports AI 工程化指南（作为实用参考，非权威证明）

链接：https://www.sports-ai.dev/blog/how-ai-predicts-football-machine-learning-guide

关键启发：

- 实用 AI 足球分析通常需要四层：多赛季历史结果、xG/底层表现、阵容与预计首发、上下文因素（主场、旅行、休息、天气、裁判、市场价格变化）。
- 可信分析应输出概率/不确定性/资料准备度，而不是单边指令。
- 赛前分析要能随着首发、天气、临场伤病和市场变化更新。

### 1.7 赔率/市场概率校准：MDPI 2025

链接：https://www.mdpi.com/2227-7390/13/24/3976

关键启发：

- 市场赔率可反推隐含概率，但会受到水位/利润率、噪声和偏差影响。
- 只看一个玩法的隐含概率不够，多个市场组之间的一致性和校准更重要。
- 本项目可展示“市场倾向/隐含概率/价格变化”，但前端要明确这不是胜率承诺，也不是操作方向。

### 1.8 FIFA Training Centre：官方 EFI xG 指标说明

链接：https://www.fifatrainingcentre.com/en/game/tournaments/fifa-womens-world-cup/2023/efi-metrics/expected-goals.php

关键启发：

- FIFA Training Centre 把 xG 作为评估单次射门进球概率的 EFI 指标，适合用于“机会质量”而不是赛果承诺。
- 对本项目来说，ESPN boxscore 的射门、射正、控球只能算基础技术统计；不能把它们包装成 xG。
- 若后续采集 FIFA/Opta/StatsBomb 级 xG，应保存供应商口径、比赛阶段、射门级上下文和采集时间，避免与基础统计混淆。

### 1.9 Opta Football Stats Definitions：xA、射门等统计定义

链接：https://theanalyst.com/articles/opta-football-stats-definitions

关键启发：

- xA 衡量传球成为助攻的可能性，适合补球员创造机会能力。
- 射门、射正、传球、对抗等基础事件需要明确定义，否则不同来源不可直接相加。
- 本项目已从 ESPN 回填比赛级射门/射正/控球，但前端应写成“基础技术指标”；只有采到 xG/xA/PPDA 等字段时才显示为“专业高阶指标”。

### 1.10 足球伤病与训练负荷机器学习研究

参考：

- 非接触伤病预测研究：https://pmc.ncbi.nlm.nih.gov/articles/PMC11694968/
- 运动机器学习综述：https://pmc.ncbi.nlm.nih.gov/articles/PMC9806754/

关键启发：

- 伤病/可用性不是简单的“有伤/无伤”，通常需要训练负荷、出场负荷、GPS/跑动、历史伤病、恢复周期和医学/训练报告。
- 当前正式库缺训练负荷、预计首发概率、恢复状态来源，因此不应把名单状态当成真实可用性。
- 第一批球员补采应优先核心球员、门将、中卫、防守型中场和定位球主罚者，并要求 `source_name/source_ref/captured_at`。

## 3. 建议补充/修改的数据

### P0：先补“能让当前前端立刻变准”的基础字段

1. 球队元数据
   - `country_iso2`、`flag_asset_key`、`confederation`、`group_name`、`metadata_source_ref`。
   - 修正 `style_tags` 中 JSON 字符串式标签，把小组信息落到 `group_name`。
   - 补 `attack_profile`、`defense_profile`、`public_sentiment` 的中文摘要。

2. 比赛基础信息
   - `kickoff_time` 必须补齐，并统一时区。
   - `venue` 可继续保留，但建议补城市、国家、球场容量、草皮/顶棚等可视字段；若暂不加表，可先进入 `external_factors`。
   - 待开球比赛要补状态时效：数据更新时间、是否确认赛程、是否需要赛前 60 分钟复核首发。

3. 球员可用性
   - 基础字段：伤病状态、停赛/红黄牌风险、训练参与、预计回归、预计首发概率、来源与更新时间。
   - 当前可先写入 `players` + `match_context_factors` + `player_metric_snapshots`，无需立即新增表。

### P1：补 AI 分析最小可用包

1. 球队高阶指标（建议每队至少 last_5 / last_10 / tournament_to_date）
   - xG、xGA、npxG、xPTS、PPDA、射门、射正、控球、渐进传球、定位球 xG、状态分。
   - 主客/中立场、赛事阶段、对手强度分层最好通过 `period_key/metric_type/raw_payload` 标明。

2. 球员高阶指标
   - 出场分钟、进球、助攻、xG、xA、npxG、射门、射正、关键传球、渐进传球、训练负荷、可用性评分、预计首发概率。
   - 重点优先核心球员、门将、中卫、防守型中场、定位球主罚者。

3. 外部因素九类覆盖
   - 天气/场地。
   - 裁判与判罚倾向（黄牌、红牌、点球、犯规尺度）。
   - 伤停与训练。
   - 赛程、旅行与休息。
   - 战意、出线形势与轮换。
   - 发布会/球队新闻/更衣室。
   - 媒体/球迷舆情。
   - 市场价格变化与公众热度。
   - 历史交锋与战术对位。

4. 市场信号
   - 现有赔率快照可继续用于隐含概率。
   - 还需要结构化 `match_market_signals`：opening/current line、opening/current odds、movement_direction、signal_level、public_bet_pct/money_pct 的中性展示口径、observed_at。
   - 重要：保存多时间点，而不是只保存一个快照，否则无法展示“变化”。

### P2：需要 OpenSpec 的结构升级候选

以下属于系统性数据模型/接口变动，若要做应先走 OpenSpec：

1. 逐射门/逐事件高阶数据
   - `match_shot_events` 或扩展 `match_events`：shot_xg、shot_xgot、shot_body_part、shot_assist_type、pressure、defender_count、keeper_position、set_piece/open_play。

2. 裁判实体库
   - 裁判主表 + 裁判赛事统计快照：近 N 场黄牌/红牌/点球/犯规/主客倾向/大赛经验。

3. 预计阵容与官方首发双轨
   - 现在 `match_lineups` 更像事实阵容；AI 赛前分析还需要 `predicted_lineups` 与 `official_lineups` 区分，并保留确认时间。

4. 赔率时间线与多公司一致性
   - 当前 `odds_market_snapshots` 可保存多快照，但前端更像目录。若要做完整走势图，需要按 market/bookmaker/observed_at 聚合并增加变化派生。

## 4. 前端需要配合修改的方向

### 4.1 总体展示原则

- 把“没有数据”拆成两类：
  1. 表/字段已支持但未入库；
  2. 当前模型尚不支持，需要后续 OpenSpec。
- 所有准备度分数都说明“不是胜率”，只是资料是否够读。
- 市场页面统一说“市场价格变化/市场倾向/隐含概率”，避免公开页出现执行指令式表达。

### 4.2 页面级建议

| 页面 | 需要补的前端表达 |
| --- | --- |
| 首页 | 增加“AI 分析数据缺口”摘要：高阶指标、球员可用性、外部因素、市场变化四条最优先 |
| 比赛中心 | 对待开球比赛显示 kickoff 缺失/已确认；完赛比分继续优先 team_stats，但事件比分兜底要标来源 |
| 赛前作战室 | 准备度从“证据数”升级为四层：基础赛程、球队高阶、球员可用性、外部/市场 |
| 舆情中心 | 当前已有覆盖矩阵，但应把空缺维度提升为补采任务卡，尤其伤停、天气、旅行、战意、市场变化 |
| 赔率中心 | 增加“单点快照 vs 时间线”提示；有 `match_market_signals` 后展示开盘-当前变化和多公司一致性 |
| 球队画像 | 国旗/小组/洲别从数据库读，不再长期依赖前端兜底；高阶指标空态要指向 `team-metrics-source.json` |
| 球员画像 | 增加“可用性先核”面板：伤病/停赛/训练/预计首发/xG-xA；空态指向 `player-availability-source.json` |
| 导入审核 | 模板校验要提示缺 `source_name/source_ref/captured_at/period_key` 的数据不要直接批准 |

### 4.3 本轮新增的公开区分口径

为避免把 ESPN 基础技术统计误称为专业高阶指标，公开球队/球员列表应区分：

- `technicalMetricCount`：球队是否已有射门、射正、控球等比赛级基础技术指标。
- `advancedMetricCount`：球队是否已有 xG、xGA、npxG、PPDA、xPTS、渐进传球、定位球 xG 等专业高阶指标。
- `performanceMetricCount`：球员是否已有进球、助攻、射门、射正、出场等比赛级基础表现指标。
- 球员侧 `advancedMetricCount`：球员是否已有 xG、xA、npxG、关键传球、渐进传球、训练负荷、可用性评分、预计首发概率等字段。

前端补采卡必须按 `advancedMetricCount` 判断 xG/PPDA 是否覆盖；不能因为 `attack_profile` 或 ESPN 基础统计已存在就显示“高阶指标已补齐”。

## 5. 推荐执行顺序

1. 先补正式库可直接承接的数据：球队元数据、kickoff_time、球队/球员 metric snapshots、外部因素、market signals。
2. 用现有模板生成 pending JSON，不直接把临时 JSON 当权威；审批入库后再验证前端。
3. 前端先做缺口显性化和任务卡，不等待所有真实数据补齐。
4. 若需要逐射门 xG、裁判实体库、预计阵容双轨、赔率走势图，再开 OpenSpec。

## 6. 第一批建议补采清单

本轮已完成 48 队基础元数据与原先 6 场待补赛果回填。后续第一批补采应转为“可审计的 AI 分析最小包”，优先覆盖仍缺高阶指标、球员可用性、外部因素和真实市场时间线的重点比赛：

1. 54 场比赛双方 xG/xGA/npxG/PPDA/xPTS、定位球 xG、渐进传球等专业高阶指标。
2. 核心球员可用性、伤停/停赛、训练参与、预计首发概率与 xG/xA。
3. 天气/场地、裁判倾向、旅行休息、战意轮换、发布会、媒体舆情、历史交锋、战术对位。
4. 市场价格变化与公众热度，至少补开盘、当前和一个中间时间点，不能只依赖单点赔率快照。
5. 球员画像事实，尤其核心球员、门将、中卫、防守型中场和定位球主罚者。

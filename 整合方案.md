## 三、SKILL.md 主流程（7 阶段）

借鉴仓库2 的 Phase 状态机，但每个阶段挂上 CLAUDE.md 的硬规则。

### 阶段 0 · 触发
- 用户提到"分析某场/某轮、要不要下注、出票、复盘"时触发。

### 阶段 1 · 官方核对（CLAUDE.md §1）
- 核对竞彩官方：比赛编号、开赛时间、玩法、让球数、是否开售。
- **HAD（普通胜平负）与 HHAD（让球胜平负）分开记录**。
- 未开售的玩法直接标记不可下注（对应 §7 巴西-海地案例）。

### 阶段 2 · 数据准备 + 多源核验（§2C，吸收仓库2 `snapshot_source`）
- 从 `data/` 取静态档案；本届已赛样本从 `data/per-match/` 取。
- 关键数据（赛果/首发/伤停/红黄牌/进球时间/控球/射正/角球/xG）**≥2 个独立来源**交叉验证。
- 每来源生成快照清单：`{url, tier(官方>数据商>媒体>预测), hash, 用途, 取数时间}` → `archive/sources/`。
- 来源冲突时写明冲突点，按更权威者处理；预测文章不得当事实。
- 证据状态标 `complete / partial / blocked`；不许有缺口却谎报完整。

### 阶段 3 · 逐场表现分析（§2B，吸收仓库1 逐场视角）
- 逐队逐场列本届本场之前全部比赛：对手/比分/进失球时间/阵型变化/关键伤停红黄/控球射门质量/定位球/反击/防守稳定性/领先落后应对。
- 判断每场结果是**实力/战术/对手失误/运气/比分掩盖问题**。
- 无本届样本 → 明确写"本届首战，无本届样本"，再补预选赛/洲际杯/历史。

### 阶段 4 · 九维 + 大赛气质分析（§2 / §2A，吸收仓库2 证据加权）
- 跑 §2 九个维度（交锋/阵容/教练/小组形势/赔率/大赛履历/比赛气质/破密集/冷门路径）。
- 证据加权：**基本面 + 盘口信号为主**，大赛气质/临场为辅；缺阵容/伤停 → 置信度降级。
- 对热门方/必须赢的队做反向审查（落后是否急躁、能否破低位、是否无效传中远射）。
- 结论必须标注是 **实力判断 / 盘口判断 / 大赛气质判断**；三者冲突时降低投入。

### 阶段 5 · 出票决策（§3 资金规则）
- 预算不固定：每轮由用户给出本轮可用资金总额；未给先问，不沿用旧 100/50。
- 单一逻辑 ≤ 总胜负资金 60%–70%；4 串 1 接受全黑风险。
- 让球深盘：让胜/让平/让负至少留两个方向保险。
- 比分分层：主比分 + 卡线比分 + 冷门比分（含 0-0/0-1/1-1/1-2 低比分路径），不单压一个比分。

### 阶段 6 · 子代理审查门（§6）
出票前按需启动：
1. 数学/规则代理（玩法、让球数、赔率乘积、金额合计）
2. 足球逻辑代理（阵容、伤病、战意、盘口风险）
3. 资金分配代理（是否过度集中、是否补保险票）
4. 大赛气质代理（热门/深盘/必须赢/历史气质争议时必启）
5. 本届逐场表现代理（已有本届样本时必启：慢热/无效控球/破密集差/守领先/低比分倾向）
6. 数据源核查代理（涉赛果/技术统计/伤停/首发/临场时必启：≥2 独立来源，标官方/媒体/预测）

### 阶段 7 · 记录 + 赛后复盘（§4 / §5，吸收仓库2 评估）
- 出票即写 `bets.json`（权威）并刷新 `archive/dashboard/index.html`；未开奖写"待赛果/待开奖"，返还盈亏留空。
- 票面 SP 优先；与网页赔率不同时备注"官网估算，票面 SP 优先"。
- 开奖后结算：返还、净盈亏、ROI。
- 五层复盘：数学层 / 足球层 / 盘口层 / 大赛气质层 / 赔率价值层（CLV）。
- **规则沉淀**：每轮至少 1 条"下次必须避免/必须增加"，回写 `rules/`（对应 §9 已沉淀的土耳其案例）。
